## 0. 지표를 어떻게 읽어야 하는가 (Answer-Aware Evaluation)

05 실험은 단순 검색(Retrieval) 지표 외에, **최종 생성된 답변(Answer)**의 품질을 함께 측정하는 지표가 추가되었습니다.

### `answer_hit_rate` (답변 적중률)
`answer_hit_rate`는 LLM이 생성한 최종 문장 안에 정답 핵심 키워드가 하나라도 포함되었는지를 측정합니다.

- 값이 높을수록: 시스템이 **최소한 방향성은 맞는 대답**을 하고 있다는 뜻입니다.
- Retrieval 지표(`hit_rate_at_k`)는 높은데 이 지표가 낮다면: 문서를 잘 찾아놓고도 LLM이 엉뚱한 문맥에 집중해서 정답을 생략했음(Lost in the middle 현상 등)을 의미합니다.

### `avg_answer_keyword_coverage` (답변 키워드 커버리지)
정답이 여러 개의 개념(예: "Planning", "Memory", "Tool use")으로 이루어져 있을 때, LLM이 그 핵심 개념들을 **얼마나 빠짐없이** 답변에 담아냈는지를 봅니다.

- 값이 `1.0`에 가까울수록: 요구하는 정답의 구성요소를 풍부하게 모두 설명했다는 뜻입니다.
- 값이 낮을수록: LLM이 오직 일부 정보만 파편적으로 제공하고 있다는 뜻입니다.

### 두 지표의 실무적 결합
현업 RAG 시스템에서는 이 두 지표의 차이를 이렇게 해석합니다.

- **`hit_rate_at_k`(검색)는 1.0인데 `answer_hit_rate`(답변)가 0.5일 때:** 프롬프트가 나쁘거나, 너무 많은 문서를 한 번에 넣어 LLM이 정답을 놓친 것입니다. (문서 재랭킹이나 프롬프트 개선 필요)
- **`answer_hit_rate`는 1.0인데 `keyword_coverage`가 0.3일 때:** 답변은 맞지만 너무 단답형이거나 내용이 부실합니다. (LLM 자체 모델을 업그레이드하거나 프롬프트에 "상세히 설명하라"는 지시 추가 필요)

## 1. 한눈에 보는 결론

가장 실질적인 최종 답변 품질을 끌어올린 설정은 **`retrieval_k_5`** 였습니다.

- `answer_hit_rate`: `0.7143` (전체 1위)
- `avg_answer_keyword_coverage`: `0.7143` (전체 1위)
- `top1_accuracy`: `0.5714` (가장 좋은 편은 아님)

이 결과는 매우 흥미롭습니다. 첫 번째 검색 순위(`top1`)가 가장 정확하지 않았음에도 불구하고, **가져오는 문서의 개수(k)를 늘렸더니 LLM이 퍼즐을 맞추듯 최종 답변을 더 완벽하게 만들어냈음**을 의미합니다.

반대로, 검색 지표(`top1_accuracy`)는 가장 높았지만 최종 답변은 부실했던 설정은 **`chunk_size_500`** 이었습니다.

- `top1_accuracy`: `0.7143` (전체 1위)
- `hit_rate_at_k`: `0.7143` (전체 꼴찌)
- `answer_hit_rate`: `0.5714` (낮은 편)

작게 자른 청크는 특정 단어를 검색해 1등으로 올리는 데는 탁월했지만, 1등을 놓쳤을 때 보완해줄 문맥이 주변에 없어 대답을 아예 못하는 비율이 가장 높았습니다.

## 2. 파이프라인 및 설정별 해석

### `retrieval_k_5` (K 늘리기: 가장 성공적인 전략)
이전 실험에서는 K를 늘리는 것이 큰 의미가 없었지만, 난이도가 올라간 이번 실험에서는 빛을 발했습니다.

- 프레임워크나 여러 구성 요소를 묻는 복합 질문(예: Agent의 Memory 종류)에서는 한두 개의 청크만으로 전체 답을 구성하기 어렵습니다.
- `k=5`로 늘리면서 파편화된 정보들이 LLM의 Context Window에 충분히 들어갔고, LLM이 이를 종합하여 가장 높은 `keyword_coverage`(0.7143)를 기록했습니다.

### `chunk_size_1500` (크게 자르기: 가장 안정적인 검색)
검색 자체의 뼈대를 튼튼하게 유지하는 데는 큰 청크가 가장 좋았습니다.

- `hit_rate_at_k`: `0.8571` (최상위권)
- `mrr`: `0.7143` (최상위권)

특정 프레임워크(예: Reflexion)의 원리를 묻는 질문일 경우, 500자나 1000자로 자르면 프레임워크 이름만 있고 설명은 다음 청크로 넘어가버릴 수 있습니다. `1500`은 이런 맥락적 단절을 막아주어 안정적인 검색을 보장했습니다.

### `baseline_agentic` (에이전트 RAG: 아직은 미완의 대기)
단순 체인(`two_step`)과 에이전트(`agentic`)를 비교한 결과입니다.

- 검색 스코어(`mrr`, `hit_rate_at_k`): `baseline_two_step`과 완전 일치함 (`0.5714` / `0.7143`)
- 답변 커버리지: `two_step`(`0.6190`) 대비 에이전트(`0.6667`)로 살짝 상승함.

현재의 질문 수준에서는 검색을 여러 번 수행해야 하는 Multi-hop 상황이 적습니다. 따라서 에이전트가 단독 직진형 체인을 압도하지는 못했습니다. 다만, LLM이 스스로 Tool의 결과물을 보고 답변을 다듬는 과정에서 키워드 커버리지가 미세하게 올랐음을 볼 수 있습니다.

## 3. 질의별 양상: 무엇이 답변을 어렵게 만드는가?

이번 05 실험을 "진짜 RAG 평가"답게 만든 일등 공신은 헷갈리는 특정 질문들이었습니다.

```python
# 현재 05_rag_tutorial_experiment_analysis.py 라인 63 부근 수정된 엄격한 질의 예시
QueryCase(
    query="What is reflection or self-criticism used for in agent systems?",
    retrieval_keywords=("reflection", "self-reflection", "self-criticism", "critique"),
    answer_keywords=("reflection", "self-criticism", "critique", "improve"),
)
```

### 1) 잘 맞는 질문 그룹 (개념 병렬형)
- Memory의 종류나 APIBank 같은 고유명사를 묻는 질문들은 검색 오류가 거의 없습니다. 
- 단어가 워낙 특이해서 문서 내에서 해당 구역 외에는 혼동할 만한 노이즈(Hard Negative)가 덜 발생하기 때문입니다.

### 2) 자주 틀리는 질문 그룹 (개념 혼동형)
- 특히 **Reflexion**이나 **Gorilla**를 묻는 질문에서 많은 모델이 실패합니다.
- 예: Reflexion의 역할을 물어봤는데, 문서 내에 있는 `Chain of Hindsight` 나 일반적인 `Self-Correction` 단락을 가장 먼저 검색물로 끌어옵니다.
- LLM 입장에서는 "비슷한 기능"처럼 보이니 그대로 답변하지만, 엄격한 키워드 매칭 기준에서는 오답 처리됩니다.
- **실무적 교훈**: 실제 RAG 서비스에서도 사용자가 "A 기능과 B 기능의 차이"처럼 특정 하위 개념을 물을 때, 엉뚱한 상위 개념을 답하는 문제가 가장 치명적입니다. 이를 해결하려면 **Metadata filtering**(태그 기반 검색)을 활용하거나, 프롬프트에서 "문맥에 명시된 이름만 사용할 것"이라고 강력히 제어해야 합니다.

## 4. 실무 파이프라인 설계 가이드 (Lesson Learned)

04 실험과 이번 05 실험의 결과를 종합해, 실제 제품(Product)을 만들 때의 의사결정 순서를 제안합니다.

1. **답변의 풍부함이 최우선인가? (예: 리포트 자동 작성, 종합 요약봇)**
   - 청크를 크게(`1500`) 자르거나
   - 가져오는 문서 개수를 넉넉히(`k=5 이상`) 늘리세요.
   - 단, 모델의 Context 허용량(Token limit)을 초과하지 않는지 대비해야 합니다.

2. **단답형의 정확성이 최우선인가? (예: 메뉴얼 봇, 규칙 검색기)**
   - 청크를 작게(`500`) 가져가서 단어 매칭의 날카로움을 살리세요.
   - 다만 이때는 첫 검색이 빗나갔을 때 "모르겠습니다"라고 깔끔하게 포기하도록 Fallback 로직을 잘 짜야 환각(Hallucination)을 막을 수 있습니다.

3. **에이전트를 도입하고 싶은가?**
   - 단순히 지식 기반 문서를 검색해서 대답하는 시스템(QA)이라면 `two_step` 체인으로 충분합니다. 비용과 속도 면에서 압도적으로 유리합니다.
   - 웹 검색, 사내 DB SQL 조회, 그리고 이 문서 검색을 **동시에 왔다 갔다 해야 하는 상황**이 오기 전까지는 `agentic` 도입을 미루는 것이 엔지니어링 낭비를 줄입니다.
